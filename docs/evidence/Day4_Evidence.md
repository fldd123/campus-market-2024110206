## 1. 今日完成内容

今天完成了发布表单的**表单校验增强**工作。在已有 `PublishView.vue` 基础上，对 `validateForm()` 函数进行了扩展，为每个字段添加了符合校园场景的现实校验规则。包括：文本字段长度限制（如标题 2~50 字符、描述 10~500 字符）、数值范围限制（价格 0.01~99999.99、目标人数 2~100、酬劳 0~9999）、联系方式格式校验（手机号 / QQ / 微信号正则匹配）、时间合理性校验（eventTime 不能晚于当前、deadline 必须晚于当前）。同时封装了 `validateLength`、`isValidContact`、`isFutureTime`、`isPastOrNow` 四个辅助函数，使校验逻辑清晰可复用。

## 2. 发布类型与字段设计

| 发布类型 | 对应数据集合 | 关键字段 | 设计理由 |
|---|---|---|---|
| 二手交易 | trades | title、category、price、condition、location、description | 需要展示商品基本信息和交易条件 |
| 失物招领 | lostFounds | title、type、itemName、location、eventTime、description | 需要区分寻物和招领，并描述物品信息 |
| 拼单搭子 | groupBuys | title、type、targetCount、deadline、location、description | 需要说明人数目标和截止时间 |
| 跑腿委托 | errands | title、taskType、reward、from、to、deadline、description | 需要说明任务内容、地点和酬劳 |

API 层使用 Axios + json-server，每类业务独立文件（`src/api/trade.ts`、`lostFound.ts`、`groupBuy.ts`、`errand.ts`），每个文件包含 `get`（获取列表）和 `create`（新增记录）两个方法，通过 `http.post<T>('/endpoint', data)` 向 `http://localhost:3001` 发送请求。`FormField.vue` 组件封装了标签展示、必填标识和错误信息插槽，供发布表单复用。

## 5. 人工调整内容

本次任务中没有需要人工额外调整的内容。AI 生成的校验规则直接符合校园场景需求。但基于项目已有设计，以下决策值得说明：

- 联系方式校验使用了三种常用格式（手机号、QQ、微信号），而非只验证手机号，因为校园场景中很多学生使用 QQ / 微信联系。
- 价格上限 99999.99 是针对校园二手交易的现实考虑（学生二手商品极少超过万元）。
- 目标人数上限设为 100 人，因为校园拼单通常是小规模组队。

## 7. 遇到的问题与解决方法

**问题：** AI 在对 `db.json` 中的 `errands` 数据分析后发现，最后一条记录的 `id` 为 `"r536PU5uXnE"`（非纯数字字符串），这是 json-server 自动生成的 id 格式。但当前的 API 接口没有在 `create` 方法中手动指定 id，json-server 会自动生成 id。

**解决：** 不需要修改代码。json-server 在 POST 请求时会自动为新增记录生成唯一 id（以字符串+随机字符的形式），无需前端手动传入 id 字段。表单提交代码中构造 `data` 对象时没有传入 `id`，这是一个正确的设计。

**其他注意事项：**
- 表单在切换发布类型时会自动调用 `resetForm()` 清除所有字段和错误消息，避免数据残留。
- 表单提交时 `submitting` 状态会禁用提交按钮并显示"提交中..."，防止重复提交。
- 由于 json-server 以 JSON 文件存储数据，并发场景下可能会有数据一致性问题，但在开发/演示阶段可以接受。

## 8. 今日反思

**发布表单**是用户与 Web 应用交互的核心入口之一，它直接决定了用户在系统中录入数据的体验和效率。一个设计良好的发布表单应当具备清晰的字段布局、合理的校验规则和及时的反馈机制。本项目中的发布表单针对四种不同业务类型（二手交易、失物招领、拼单搭子、跑腿委托）分别展示不同的字段集合，通过 `v-if` 条件渲染让用户只看到当前类型需要的字段，避免了信息过载。

**表单校验**是保证数据质量的第一道防线。如果没有校验，用户可能会提交空标题、负数价格、不合法的联系方式等脏数据，导致下游列表页面渲染异常或数据统计错误。本次新增的校验规则覆盖了长度、格式、范围和时间四类约束，将错误在提交前就拦截下来，并给出具体的错误提示，帮助用户快速修正。

**数据新增**是 CRUD 中 Create 操作的体现，它通过 API 层向 mock 服务发送 POST 请求，将表单数据持久化到 `db.json` 文件中。与简单的"保存到变量"不同，数据新增涉及数据模型定义、HTTP 请求、服务端存储、页面跳转和用户反馈等多个环节，任何一个环节出错都会导致"发布失败"的糟糕体验。

通过这三者的有机结合——发布表单收集数据、表单校验保障质量、数据新增完成持久化——一个完整的"发布信息"功能才得以闭环运转，这也是 Web 应用中"增删改查"中最基础也最重要的一环。
